Glossary

Machine Learning & Statistics Terminology
Evaluation Metrics
Mean Absolute Error MAE Metrics
The average of absolute differences between predicted and actual values. Expressed in the same units as the target (dollars for revenue prediction). Lower is better.
MAE = (1/n) * Σ|predicted - actual|
In this app: A MAE of $50 means predictions are off by $50/month on average. Reported for both XGBoost and Neural Network regression models.
Mean Absolute Percentage Error MAPE Metrics
The average of absolute percentage differences between predicted and actual values. Unlike MAE, it provides a scale-independent measure of prediction accuracy. Can be undefined when actual values are zero.
MAPE = (1/n) * Σ(|predicted - actual| / |actual|) * 100%
In this app: A MAPE of 15% means predictions are off by 15% of the actual revenue on average. Shown alongside MAE in regression results.
Symmetric Mean Absolute Percentage Error SMAPE Metrics
A percentage-based error metric that treats over-predictions and under-predictions symmetrically. Ranges from 0% (perfect) to 200% (worst). Unlike MAPE, it doesn't explode when actual values are near zero.
SMAPE = (1/n) * Σ(|pred - actual| / ((|actual| + |pred|) / 2)) * 100%
In this app: A SMAPE of 30% means predictions deviate by ~30% of the average of predicted and actual revenue.
Root Mean Squared Error RMSE Metrics
The square root of the average squared differences between predicted and actual values. Penalizes large errors more heavily than MAE. Expressed in the same units as the target.
RMSE = √((1/n) * Σ(predicted - actual)²)
In this app: If RMSE is much larger than MAE, some sites have very large prediction errors (outliers).
Coefficient of Determination R² Metrics
Measures how much variance in the target the model explains, compared to simply predicting the mean. Ranges from -∞ to 1.0 where 1.0 is perfect and 0 means "no better than guessing the average."
R² = 1 - (Σ(actual - pred)² / Σ(actual - mean)²)
In this app: R² of 0.7 means the model explains 70% of the variance in site revenue. The primary metric for comparing regression experiments.
Area Under the ROC Curve ROC-AUC Metrics
Measures how well the model distinguishes between positive and negative classes across all possible thresholds. Ranges from 0.5 (random guessing) to 1.0 (perfect separation). Used for classification tasks.
In this app: The primary metric for lookalike models. Measures how well the model identifies top-performing sites vs. others.
F1 Score F1 Metrics
The harmonic mean of precision and recall. Ranges from 0 to 1. It balances the trade-off between catching all positive examples (recall) and avoiding false positives (precision). More informative than accuracy for imbalanced datasets.
F1 = 2 * (precision * recall) / (precision + recall)
In this app: Reported alongside AUC for lookalike classification. A high F1 means the model both finds most top performers and doesn't misclassify too many non-performers.
Silhouette Score Metrics
Measures how similar a data point is to its own cluster compared to neighboring clusters. Ranges from -1 (wrong cluster) to +1 (well-separated clusters). 0 indicates overlapping clusters.
In this app: The primary metric for evaluating Deep Embedded Clustering (DEC) results. Higher scores mean more distinct performer segments.
Loss Metrics
The objective function that the model minimizes during training. Lower loss means better predictions. For regression, this app uses Huber Loss (robust to outliers). For classification, it uses Binary Cross-Entropy Loss. For clustering, it uses a combination of reconstruction loss and KL-divergence.
In this app: Training loss should decrease over epochs. Validation loss that stops decreasing signals overfitting.
Accuracy Metrics
The proportion of correct predictions out of total predictions. For classification only. Can be misleading with imbalanced classes (e.g., 90% accuracy by always predicting the majority class).
Accuracy = correct predictions / total predictions
In this app: Used alongside AUC and F1 for the lookalike model. AUC is the primary metric since classes are imbalanced (90/10 split).
Brier Score Metrics
Measures the accuracy of probabilistic predictions. Calculated as the mean squared difference between predicted probabilities and actual binary outcomes. Ranges from 0 (perfect) to 1 (worst). Sensitive to calibration.
Brier = (1/n) * Σ(predicted_prob - actual)²
In this app: Used to evaluate probability calibration before and after applying isotonic regression or Platt scaling.
Training Parameters
Epoch Training
One complete pass through the entire training dataset. Multiple epochs allow the model to refine its parameters iteratively. Too many epochs can lead to overfitting; too few leads to underfitting.
In this app: Default is 50 epochs for neural networks. XGBoost uses "boosting rounds" which serve a similar purpose. Early stopping may halt training before the limit.
Batch Size Training
The number of training examples processed together before updating model weights. Larger batches provide more stable gradient estimates but use more GPU memory. Smaller batches add noise that can help escape local minima.
In this app: Default is 4096, optimized for Apple Silicon GPU (MPS) memory bandwidth. Only applies to neural network training; XGBoost processes data differently.
Learning Rate LR Training
Controls how much model weights are adjusted in response to each batch's error gradient. Too high causes instability (loss oscillates or diverges). Too low causes extremely slow convergence.
In this app: Default is 0.0001 (1e-4) for neural networks. XGBoost uses a separate learning rate (eta) that controls the contribution of each tree. A scheduler automatically reduces LR when validation loss plateaus.
Dropout Training
A regularization technique that randomly sets a fraction of neuron outputs to zero during training. This prevents the network from relying too heavily on any single neuron, reducing overfitting.
In this app: Default is 0.2 (20% of neurons deactivated per layer during training). Only applies to neural networks, not XGBoost. Only active during training, not prediction.
Early Stopping Training
Halts training when validation performance stops improving for a specified number of epochs (the "patience"). Prevents overfitting by keeping the model from memorizing training data noise.
In this app: Patience is 10 epochs for neural networks and 10 boosting rounds for XGBoost. The best model checkpoint is restored after stopping.
Weight Decay Training
A regularization technique that penalizes large model weights by adding a fraction of the weight magnitude to the loss. Encourages simpler models that generalize better.
effective_loss = loss + weight_decay * Σ(w²)
In this app: Default is 1e-5 for neural networks. XGBoost uses separate L1/L2 regularization parameters (alpha/lambda).
Learning Rate Scheduler Training
Automatically adjusts the learning rate during training. This app uses "Reduce on Plateau" — when validation loss stops improving for a set number of epochs (scheduler patience), the learning rate is halved.
In this app: Scheduler patience is 5 epochs. The LR is multiplied by 0.5 each time the plateau is detected. Only applies to neural network training.
Model Architecture
XGBoost Extreme Gradient Boosting Model
A gradient boosting framework that builds an ensemble of decision trees sequentially. Each new tree corrects errors from prior trees. Known for speed, accuracy, and built-in regularization. Uses histogram-based split finding for efficiency on large datasets.
In this app: Available for regression and as one of two model types. Uses the 'hist' tree method optimized for CPU. Automatically handles missing values and feature interactions without embeddings. Generally faster to train than neural networks but may underperform on high-cardinality categorical features.
Neural Network Model
A machine learning model inspired by biological neurons. Composed of layers of interconnected nodes that transform input features through learned weights and non-linear activations to produce predictions.
In this app: A deep feedforward network with configurable hidden layers (default: 512 → 256 → 128 → 64 neurons), residual connections, and categorical embeddings. Available for all task types: regression, lookalike classification, and clustering. Runs on Apple Silicon GPU (MPS) for acceleration.
Deep Embedded Clustering DEC Model
An unsupervised deep learning approach that jointly learns feature representations and cluster assignments. Combines an autoencoder (for learning a compact latent space) with learnable cluster centroids. Uses two training phases: pretraining (reconstruction) and clustering refinement via KL-divergence.
In this app: Used for the "Cluster Performers" task to discover natural groupings among high-performing sites. Soft cluster assignments are refined using Student's t-distribution, and sites are assigned to clusters based on a configurable probability threshold.
Gradient Boosting Model
An ensemble technique that builds models sequentially, where each new model attempts to correct the residual errors of the combined ensemble so far. The "gradient" refers to using gradient descent in function space to optimize a loss function. Decision trees are the most common base learner.
In this app: XGBoost is the gradient boosting implementation. Each boosting round adds a new tree that focuses on examples the ensemble currently gets wrong.
Hidden Layers Model
The intermediate layers between input and output in a neural network. Each layer applies a linear transformation followed by a non-linear activation function. More/wider layers increase capacity but also risk overfitting.
In this app: Default is [512, 256, 128, 64] — a progressively narrowing architecture that forces the model to learn compact representations. Configurable via the hyperparameters panel.
Residual Connections Skip Connections Model
Shortcut paths that add a layer's input directly to its output, allowing gradients to flow more easily through deep networks. Helps prevent vanishing gradients and enables training of deeper architectures.
In this app: The neural network uses residual blocks where each hidden layer's output is added to its input (when dimensions match). This improves training stability for the 4-layer architecture.
Embedding Dimension Model
The size of the dense vector representation learned for each categorical feature value. Embeddings capture relationships between categories (e.g., similar networks have similar embeddings) in a continuous space.
In this app: Each categorical feature (network, program, retailer, etc.) gets a 16-dimensional embedding vector. Only used by neural networks; XGBoost handles categoricals natively via label encoding.
Batch Normalization Model
Normalizes the output of each layer to have zero mean and unit variance. This stabilizes training, allows higher learning rates, and acts as mild regularization.
In this app: Applied after each hidden layer in the neural network. Helps the model train faster and more reliably on the diverse site features.
Regression Model
A task where the model predicts a continuous numeric value. The model outputs a single unbounded number representing the predicted quantity.
In this app: "Predict Revenue" task. Predicts average monthly revenue per site. Available with both XGBoost and Neural Network. Uses Huber Loss and reports MAE, MAPE, SMAPE, RMSE, R².
Classification (Lookalike) Model
A task where the model predicts which class an example belongs to. In binary classification, the model outputs a probability of belonging to the positive class (top performers).
In this app: The "Find Lookalikes" task identifies sites that resemble top revenue performers. Uses configurable percentile bounds (default: p90-p100). Neural network only. Reports AUC, F1, and accuracy.
Clustering Model
An unsupervised task where the model discovers natural groupings in data without labeled examples. Sites are assigned to clusters based on feature similarity, revealing hidden segments in the portfolio.
In this app: The "Cluster Performers" task uses Deep Embedded Clustering (DEC) to segment high-performing sites into distinct groups. Configurable number of clusters (default: 5) and assignment confidence threshold.
Huber Loss Model
A loss function that is quadratic for small errors and linear for large errors. This makes it less sensitive to outliers than pure squared error (MSE), while still being smooth and differentiable everywhere.
In this app: Used for regression with delta=1.0. Handles the fact that a few sites have extreme revenue values without distorting the model.
Binary Cross-Entropy Loss BCE Model
The standard loss function for binary classification. Measures how far predicted probabilities are from actual binary labels (0 or 1). Penalizes confident wrong predictions more heavily than uncertain ones.
In this app: Used with pos_weight for the lookalike task to compensate for class imbalance (only ~10% of sites are top performers).
KL-Divergence Kullback-Leibler Model
A measure of how one probability distribution diverges from a second expected distribution. In clustering, it drives soft cluster assignments toward a sharper target distribution, encouraging the model to make confident cluster assignments.
In this app: Used as the clustering refinement loss in DEC. Minimizing KL-divergence between soft assignments and an auxiliary target distribution sharpens cluster boundaries.
Class Imbalance Weighting pos_weight Model
A correction applied when one class is much rarer than the other. Multiplies the loss for the minority class by a weight so the model doesn't learn to always predict the majority class.
In this app: Automatically computed based on the percentile bounds. For p90-p100 (top 10%), pos_weight ≈ 9.0 — each positive example counts ~9x in the loss calculation.
Data & Processing
Training Set Data
The portion of data used to train the model (learn weights). The model sees these examples repeatedly during training and optimizes its parameters to minimize prediction error on them.
In this app: 70% of active sites (~18,270 sites). Displayed as blue dots on the map.
Validation Set Data
Data used to monitor performance during training without training on it. Used for early stopping decisions and learning rate scheduling. Helps detect overfitting (train loss decreasing while val loss increases).
In this app: 15% of active sites. Metrics shown during training (val_loss, val_mae, etc.) come from this set.
Test Set Data
Data held out entirely until after training completes. Provides an unbiased estimate of final model performance. Never used for any training decisions.
In this app: 15% of active sites. Displayed as cyan dots on the map. Final reported metrics (Test MAE, Test R², etc.) come from this set.
Feature Scaling StandardScaler Data
Transforms numeric features to have zero mean and unit standard deviation. Essential for neural networks because features on different scales (e.g., latitude vs. revenue) would otherwise dominate gradient updates unevenly. XGBoost is scale-invariant and doesn't require this step, but it's applied uniformly.
scaled = (value - mean) / std_deviation
In this app: Applied to all numeric features. The scaler is saved in the experiment folder as preprocessor.pkl for consistent inference.
Feature Types Data
Features are categorized into three types: Numeric (13 default) for continuous variables like distances and income; Categorical (7 default) for discrete labels like network and retailer; and Boolean (40 default) for binary flags like EMV-enabled or NFC-capable.
In this app: You can select individual features from each type via the tabbed feature selector on the home page. The feature count badge shows selected vs. available.
Data Leakage Data
When information about the target variable accidentally appears in the input features. This inflates metrics during training but produces useless predictions on new data. A common and dangerous pitfall in ML.
In this app: Revenue-derived features (total_revenue, revenue_per_screen, etc.) are automatically removed when predicting revenue to prevent leakage.
Percentile Clipping Data
Capping extreme values at specified percentiles (e.g., 1st and 99th) to reduce the influence of outliers. Values below P1 are set to P1, values above P99 are set to P99.
In this app: Both features and targets are clipped to [P1, P99]. Prevents a few extreme-revenue sites from distorting the model.
Percentile Bounds (Lookalike) Data
Configurable thresholds that define the "top performer" positive class for the lookalike model. Sites with revenue at or above the lower percentile and at or below the upper percentile are labeled as positive (1).
In this app: Default is p90-p100 (top 10%). Configurable via the home page. Narrower ranges (e.g., p95-p100) create a more selective model; wider ranges (e.g., p80-p100) are more inclusive.
Network Subset Data
A filter that restricts training and prediction to sites belonging to a specific network. This allows building specialized models for different network segments of the portfolio.
In this app: Available networks are All Networks (26,099 sites), Gilbarco (18,210), Speedway (2,760), and Wayne/Dover (5,129). Selected on the home page before training.
Overfitting Data
When a model memorizes training data noise rather than learning generalizable patterns. Symptoms: training loss keeps decreasing but validation loss starts increasing. Remedies include dropout, early stopping, weight decay, and more data.
In this app: Monitored by comparing train_loss vs val_loss each epoch in the training charts. Early stopping, dropout (NN), and regularization (XGBoost) are the primary defenses.
Site Scores Dataset Data
The primary source dataset containing 1.47 million monthly performance records across 57,675 sites with 94 columns. Each row represents one site's performance for one month, including revenue, impressions, diagnostics, demographics, capability flags, and restriction flags.
In this app: Located at data/input/site_scores_revenue_and_diagnostics.csv (927 MB). All other datasets are derived from or joined to this source. The status column is named statuis (typo in source data).
Site Aggregated Dataset Data
Intermediate dataset with all 57,675 sites aggregated from monthly records to one row per site. Created by the ETL pipeline, it sums revenue and impressions, computes averages, and joins geospatial features (nearest site distance, interstate distance, retailer distances).
In this app: Located at data/processed/site_aggregated_precleaned.parquet. Used by get_all_sites_for_prediction() to score all sites during batch inference.
Training Dataset Data
The ML-ready dataset filtered to 26,099 Active sites with 12+ months of history and positive revenue. This is the data actually fed into model training after the 70/15/15 train-validation-test split.
In this app: Located at data/processed/site_training_data.parquet. Generated by prepare_training_dataset(active_only=True) in the ETL pipeline.
Monthly Site Activity Data
An intermediate parquet file that merges daily transaction counts and daily status snapshots into monthly aggregates per site. Contains columns for active days, transaction days, and total days observed in each month.
In this app: Located at data/processed/site_monthly_activity.parquet (83,874 rows, 6 columns). Joined into the aggregated dataset to provide operational consistency features like active_days and transaction_days.
Retailer Distance Features Data
Pre-computed minimum distances from each site to the nearest Kroger, McDonald's, Walmart, and Target location. These serve as proxies for foot traffic potential and commercial area density — sites near major retailers tend to have different revenue profiles.
In this app: Four CSV files in data/input/ (site_kroger_distances.csv, site_mcdonalds_distances.csv, site_walmart_distances.csv, site_target_distances.csv), each with 57,675 rows. Log-transformed and joined during ETL.
ETL Summary Reports Data
Human-readable text reports generated by the ETL pipeline after each run. They summarize dataset dimensions, status distributions, column breakdowns, and key statistics — useful for verifying pipeline output without loading the full parquet files.
In this app: Two files in data/processed/: precleaned_summary.txt (covers all 57,675 sites) and training_data_summary.txt (covers 26,099 Active training sites). Regenerated on each ETL run.
Experiment Folder Data
A self-contained directory of artifacts from a single model training run. Each folder includes the saved model checkpoint, preprocessor (scaler + encoders), full configuration, SHAP feature importance, test predictions, and training site list — everything needed to reproduce or deploy that model.
In this app: Stored at site_scoring/outputs/experiments/job_*/. Limited to 10 folders (FIFO cleanup). Key files: config.json, best_model.pt (NN) or model_wrapper.pkl (XGB), preprocessor.pkl, shap_importance.json.
Model Interpretability
SHAP SHapley Additive exPlanations Interpretability
A game-theoretic approach to explain the output of any machine learning model. SHAP assigns each feature an importance value (Shapley value) for a particular prediction, showing how much each feature pushes the prediction above or below the average.
In this app: After training completes, SHAP values are computed on test samples to identify which site features most influence revenue predictions or lookalike scores. Works with both Neural Network (KernelExplainer) and XGBoost (TreeExplainer) models.
SHAP Value Interpretability
The contribution of a single feature to the difference between the actual prediction and the average prediction (base value). Positive SHAP values push the prediction higher; negative values push it lower. The sum of all SHAP values equals the difference between the model's prediction and the base value.
prediction = base_value + Σ SHAP_values
In this app: A positive SHAP value for "screen_count" means more screens pushes the predicted revenue higher for that site.
Probability Calibration Interpretability
The process of adjusting model-output probabilities so they reflect true observed frequencies. A well-calibrated model that predicts 80% probability should be correct ~80% of the time. Methods include Isotonic Regression (non-parametric) and Platt Scaling (logistic fit).
In this app: Applied to lookalike classification output. Calibrated probabilities are essential for the tier classification system and for meaningful probability thresholds in business decisions.
Conformal Prediction Interpretability
A framework that provides prediction sets with statistical coverage guarantees. Instead of a single prediction, it outputs a set of possible classes with a guaranteed probability of containing the true label. Based on split conformal inference using non-conformity scores.
In this app: For lookalike classification, conformal prediction wraps the model output to give confidence-aware prediction sets. If the set contains both classes, the model is uncertain about that site.
Counterfactual Explanations Interpretability
"What-if" explanations that show the smallest changes to input features that would flip the model's prediction. They answer: "What would need to change for this site to be classified as a top performer?" Only actionable features (not location or demographics) are modified.
In this app: Uses DiCE (Diverse Counterfactual Explanations) to generate actionable recommendations. Immutable features (latitude, household income) are locked; actionable features (hardware, payment tech) can vary.
Tier Classification Interpretability
A business-friendly translation of calibrated model probabilities into actionable tiers. Maps continuous probability scores to discrete recommendation levels, making model output interpretable for non-technical stakeholders.
In this app: Tier 1 "Recommended" (>85%), Tier 2 "Promising" (65-85%), Tier 3 "Review" (50-65%), Tier 4 "Not Recommended" (<50%). Each tier includes recommended actions and historical accuracy context.
Base Value Expected Value Interpretability
The average model output across the background dataset. This is the prediction the model would make if it had no information about a specific sample's features. SHAP values represent deviations from this baseline.
In this app: The base value represents the average predicted revenue (or lookalike score) across the background sample of ~100 sites.
Feature Importance (Global) Interpretability
The mean absolute SHAP value for each feature across all explained samples. Features with higher mean |SHAP| have a greater overall impact on model predictions.
importance(feature) = mean(|SHAP_values|) across all samples
In this app: The bar chart on the SHAP Values page ranks all features by their average impact. For XGBoost, native feature importance (gain) is also available as an alternative ranking.
Beeswarm Plot Summary Plot Interpretability
A visualization where each dot represents one sample's SHAP value for one feature. The x-axis shows the SHAP value (impact on prediction), and color indicates the feature value (red = high, blue = low). This reveals both importance and the direction of effect.
In this app: If "avg_household_income" shows red dots (high income) on the right (positive SHAP), it means higher-income areas predict higher revenue.
KernelExplainer Interpretability
A model-agnostic method for computing SHAP values that works with any prediction function. It approximates Shapley values by sampling feature coalitions and measuring how predictions change. Slower than tree-specific methods but compatible with neural networks.
In this app: Used for neural network models. Computes SHAP on ~200 test samples using 100 background samples for efficiency.
TreeExplainer Interpretability
An exact, polynomial-time algorithm for computing SHAP values on tree-based models (XGBoost, Random Forest, etc.). Much faster than KernelExplainer because it exploits the tree structure to compute exact Shapley values without sampling.
In this app: Automatically used when the model is XGBoost. Computes exact SHAP values for all features across all test samples.
Background Data Interpretability
A representative subset of training data used by the SHAP explainer to define the "expected" model behavior. SHAP measures each feature's contribution by comparing predictions with and without that feature, using background values as replacements.
In this app: 100 randomly sampled test instances are used as background. More samples increase accuracy but also increase computation time.
Shapley Values Interpretability
A concept from cooperative game theory that fairly distributes a "payout" (prediction) among "players" (features). Each feature's Shapley value is its average marginal contribution across all possible orderings of features. SHAP values are the ML application of this concept.
Properties: Shapley values are the only attribution method satisfying efficiency (values sum to prediction), symmetry, and linearity simultaneously.
